Four leaf files form the application's top-level bootstrap and cross-cutting concerns:
- `app.js` is the process entry point: it loads `.env`, connects to MongoDB via Mongoose, configures EJS + ejs-mate view engine, static assets, `express-session` with `connect-mongo` persistence (7-day cookie, httpOnly, secure in production), `connect-flash`, `cookie-parser`, and Passport LocalStrategy wired against `models/user`. It registers global res.locals (`success`, `error`, `currUser`) and mounts three route modules under `/listings`, `/listings/:id/reviews`, and `/`. A catch-all handler delegates 404s to a custom `ExpressError`, which the subsequent error-handling middleware renders as an EJS template.
- `middleware.js` exports reusable Express middleware: authentication gate (`isLoggedin`), owner-check (`isOwner` on Listing), review-authority check (`isreviewAuthor` on Review), redirect-url preservation (`saveRedirectUrl`), and request-body validators (`validateListing`, `validateReview`) that delegate to Joi schemas from this module's `Schema.js` and throw `ExpressError(400)` on failure.
- `Schema.js` is a pure Joi schema registry exporting `listingSchema` and `reviewSchema`; it has no runtime side effects and is consumed only by `middleware.js`.
- `cloudConfig.js` initializes Cloudinary v2 from env vars and builds a `multer-storage-cloudinary` storage adapter targeting the `wanderlust_DEV` folder; it is imported by file-upload routes elsewhere.

Dependency direction is one-way outward: `app.js` → `middleware.js` / `Schema.js` / `cloudConfig.js` / route modules / models; `middleware.js` → `Schema.js` + models; `cloudConfig.js` and `Schema.js` are leaf utilities with no internal dependencies.
